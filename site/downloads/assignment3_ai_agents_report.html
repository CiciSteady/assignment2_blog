Online Agent: DeepSeek Q&A Platform

The online agent is implemented as site/deepseek_qa.html with JavaScript in site/assets/deepseek_qa.js. The page asks the user to enter a DeepSeek API Key in the browser. The key is not hard-coded in the source files, which avoids exposing secrets when uploading the project to GitHub.

The JavaScript request uses the DeepSeek chat-completions endpoint:

POST https://api.deepseek.com/chat/completions
Authorization: Bearer <DeepSeek API Key>
Content-Type: application/json

The page supports model selection for deepseek-v4-flash, deepseek-v4-pro, and legacy compatibility options such as deepseek-chat and deepseek-reasoner. It also includes a context text area and a text-file upload input, so the agent can answer questions based on pasted course notes, code, Markdown, or other text files.

Local Model Deployment with Ollama

The local deployment workflow uses Ollama. The intended commands are:

ollama pull qwen2.5:7b
ollama serve
python scripts\ollama_local_agent.py --model qwen2.5:7b --prompt "Explain what an AI coding agent does in three bullet points."

In the checked workspace, Ollama was not running, so the local script prints setup guidance instead of failing silently. This makes the deployment process reproducible once Ollama is installed.

Challenges and Solutions

  • API key security: the key is entered in the browser and not committed to GitHub.
  • Static GitHub Pages limitation: the page is front-end only; if a browser blocks API requests, a local proxy or Python script can be used for demonstration.
  • Ollama availability: the local script reports clear installation and model-pull steps when the service is unavailable.
  • Workflow verification: Python scripts were syntax-checked and refactor examples were run before submission.

Reflection

Online models are the fastest way to build a useful agent because the model capability is provided by the API. They are suitable for complex reasoning, file analysis, and code explanation. The main concern is privacy and API-key management.

Local models require more setup, but they give better control over privacy and offline use. A practical workflow is to use online models for difficult reasoning and local models for private drafts or experiments. IDE integration is most useful when the prompt is specific and the result is verified with tests or command-line checks.
